所見の詳細

01

行頭でBackspaceすると、本文が無関係な枝の奥に紛れ込む

重要度 高
Z3 · flat_order_merge.py getFlatOrder backspaceAtStart

宣言

行頭Backspaceは、その行をひとつ前の行に連結する（アウトライナーの常識）。

コードの実際

連結先を「木の隣（親・兄弟）」ではなくDFS画面順で1つ前のノードから選ぶ。DFSの1つ前は、しばしば別の枝のいちばん深いノード。

Z3が自動発見した反例 — Bの行頭でBackspace

ルート                         ルート
├─ A            ─Backspace→    ├─ A
│   └─ A1                      │   └─ A1"B"   ← Bの本文がここに合体
└─ B  ←ここ                    ├─ B1     ← 子はルート直下へ散逸
    ├─ B1                      └─ B2
    └─ B2
merge先 = A1（親=A）／ B の親 = ルート → 親でも兄弟でもない

意図：「Bの行を消して前とつなげたい」。結果：Bの本文は無関係なA1に紛れ込み、Bの子は本文と切り離されてルート直下に散らばる。カーソルも別の枝へ飛ぶ。連結先は折りたたみ状態にも依存する。

修正済み → mergeIntoPredecessor（前の兄弟、無ければ親へ連結。子は本文と一緒に移動）と対称の mergeSuccessorInto（最初の可視の子／次の兄弟のみ引き上げ、木構造上の後続が無ければ no-op）を新設し、backspaceAtStart / deleteAtEnd をこれらに切替。

02

読込時に「木であること・IDが一意であること」を検査していない

重要度 高
Alloy · MindMap.als parseContent findNode / removeNode

宣言

MindMapModel は「木」。ドメイン層はIDがノードを一意に指す前提（findNode・cloneWithNewIds の存在自体が証拠）。

コードの実際

parseContent は id:string / text:string / children:array の形だけ検査。一意性・単一親・非循環・木かどうかは検査しない。入力はDB／API経由の外部JSON。

Alloyが発見 — 形の検査を通る「木に見える」重複ID構造

ルート(id=dup)
├─ 子(id=dup)   "First"
└─ 子(id=dup)   "Second"

removeNode("dup")  →  "First" だけ消え "Second" が残る
findNode("dup")    →  残った別ノード "Second" を返す
他に SharedChild（多重親=DAG）・循環構造 も受理されることを確認

影響：「ノード単位のWeb公開」（TODO）や画像アップロード先の指定はIDでノードを一意に指すことに依存する。壊れた／悪意あるJSONを1つ保存されると、以後その文書のあらゆる編集が「最初の一致だけ」に効いて予測不能になる。

修正済み → persistence.ts に normalizeTree() を新設し、parseContent がルートの形だけでなく木全体を深さ優先で検証。欠落／非文字列／既出のIDを新IDへ再採番して一意IDの木を保証し、形をなさない子ノードは除去（既知の任意フィールドは型チェックの上で保持）。

03

Undo/Redo後、選択が「存在しないノード」を指したままになりうる

重要度 中
TLA+ · MindMapUndo.tla Z3 · undo_redo.py reconcileView

宣言

「常に1ノードが選択され、activeNodeId は null にならない」。Undoは「文書だけ復元、フォーカスは動かさない」。

コードの実際

型は string | null。「選択が生きたノードを指す」不変条件は、呼び出し側が別途 reconcileView() を呼ぶことで回復される規約であって、型・状態としては強制されていない。

TLA+ が発見した反例トレース（ReconcileAuto = FALSE）

Init      : 文書={ルート}        選択=ルート
AddNode   : 文書={ルート, n1}     選択=n1
Undo      : 文書={ルート}        選択=n1  ← 文書に存在しない
            ⇒ INVARIANT ActiveAlwaysLive 違反
Z3(part B)でも同一状態を発見。Undo/Redoの代数自体は健全（part A）。
ReconcileAuto=TRUE に切替えると不変条件は保たれる ＝ 修正案。

影響：reconcile前の隙に選択が亡霊ノードを指すと、findNode が null を返しキーボード操作が全て静かに無反応になる。同種の無反応バグは過去に修正実績あり（コミット a6def5e）。

修正済み → Undo/Redo が通る editorReducer の replace パスに reconcileView() を畳み込み、「選択は生きたノードを指す」不変条件を reducer 自身が強制（冪等）。restoreDocument は事前 reconcile を廃止。型の非null化は影響範囲が広く今回は見送り。

04

行頭でEnterすると、ノードの同一性（ID）が入れ替わる

重要度 中
実行プローブ splitNode

子を持つノードの先頭でEnter

P("Parent")            P("")            ← 元ノード(元ID)は空になり子を失う
└─ C("Child")   →      新ID("Parent")
                       └─ C("Child")     ← 本文と子は新IDへ移動

影響：純粋なテキストアウトライナーなら「上に空行が入った」ように見えて自然だが、edaneはノードIDを外部から参照する設計（画像先・リンク・ノード単位公開）。「行を1つ足しただけ」で本文の載るIDが変わり、そのノードを指す外部URLが空ノードを指しうる。

修正済み → splitNode の行頭（pos≤0）分割を、元ノードの ID・本文・子をそのまま残し空の兄弟を1つ上に挿入する挙動へ変更。enter も pos≤0 でカーソルを元ノードに残す。外部参照が空ノードを指すことはなくなった。

方法についての注記

各性質は「宣言仕様」「コード仕様」の二つを別々に形式化し、両者が食い違う具体例をソルバに探索させた。Alloyは4ノード、Z3は6ノードの有界スコープで探索している（有界モデル検査）。反例は実装の再実装ではなく本物のドメインコードに対して回帰テスト化済み（formal/repro/reproduce.test.ts、修正後の挙動を固定して5件パス）。

TLA+ の TLC はこのサンドボックス（外向き通信が PyPI / npm / Maven Central に限定）から tla2tools.jar を取得できず未実行。TLA+ が主張する選択不変条件は、実行可能な Z3 モデル（undo_redo.py）で同一の反例をクロスチェック済みで、仕様は標準構文のためローカルの TLC でそのまま検査できる。
