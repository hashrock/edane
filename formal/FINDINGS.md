# 形式検証で見つかった仕様の綻び — ドメイン説明

対象: edane（テキストのインデントからマインドマップを生成するアウトライナー）
方法: ドキュメント／コードコメントが**宣言している仕様**と、コードが**実際に計算している仕様**をそれぞれ Z3・Alloy・TLA+ で形式化し、モデル検査で両者が食い違う具体例（反例）を機械的に探索した。
再現: すべての反例を実コードに対して `formal/repro/reproduce.test.ts` で再現テスト化済み（4件パス）。

各所見は「マインドマップ／アウトライナーの利用者・ドメインエキスパートにどう見えるか」で説明する。

## 所見 1 — 行頭でBackspaceすると、本文が**無関係な枝の奥**に紛れ込む【重要度：高】

**モデル**: `formal/z3/flat_order_merge.py`（Z3が反例の木を自動発見）
**再現**: FINDING 1（`reproduce.test.ts`）

### 何が宣言されているか
- `getFlatOrder()` は「ナビゲーション順（DFS順）」＝画面上の並び順。
- `backspaceAtStart`（行頭でBackspace）は「その行を**ひとつ前の行**に連結する」というアウトライナーの常識的な操作。

### コードが実際にやること
連結先を「木構造の隣（親や兄弟）」ではなく「**DFSの画面順で1つ前のノード**」から選ぶ。DFSの1つ前は、しばしば**別の枝のいちばん深いノード**になる。

### 反例（ドメインの言葉で）
```
ルート
├─ A
│   └─ A1
└─ B          ← ここ（"B"の行頭）でBackspace
    ├─ B1
    └─ B2
```
利用者の意図：「Bの行を消して前とつなげたい」。
実際の結果：

```
ルート
├─ A
│   └─ A1"B"      ← Bの本文が、隣の枝の2段深いノードA1に合体
├─ B1            ← Bの子は、ルート直下へ昇格して散らばる
└─ B2
```

**Bの本文は関係のないA1に紛れ込み、Bの子（B1・B2）は本文と切り離されてルート直下に散らばる。** カーソルも別の枝（A1）へ飛ぶ。1回のBackspaceで、意味的にまとまっていた「見出しとその小項目」がバラバラの場所に移動する。

Z3は6ノードの探索で、より極端な連鎖形（`ルート→B→C→D→E` の直系列 ＋ ルート直下のF）も自動発見した：Fの行頭Backspaceで、Fの本文が4段深いEに合体する。

### ドメイン的な影響
アウトライン編集で最も多用されるキー操作のひとつが、**内容を静かに別の場所へ移してしまう**。しかも移動先は画面レイアウト（折りたたみ状態含む）に依存するため、同じ木でも折りたたみ次第で挙動が変わる。

### 直し方の方向性
連結先を「DFSの1つ前」ではなく「同じ深さの前の兄弟」または「親」に限定する（本文を持つ祖先/兄弟が無いときは連結しない＝no-op）。同じ選び方をしている `deleteAtEnd`（行末Delete）も対称に見直す。

## 所見 2 — 読み込み時に「木であること・IDが一意であること」を検査していない【重要度：高】

**モデル**: `formal/alloy/MindMap.als`（Alloyが受理形状の反例を発見）
**再現**: FINDING 2（`reproduce.test.ts`）

### 何が宣言されているか
- README／型定義：`MindMapModel` は「木（tree）」。
- ドメイン層全体が「IDはノードを一意に指す」前提で書かれている：
  - `findNode(id)` … そのIDのノード「1つ」を返す
  - `findParentAndIndex(id)` … 親「1つ」と位置を返す
  - `cloneWithNewIds()` … 貼り付け時にIDが衝突しないよう**わざわざ**新IDを振る関数が存在する

### コードが実際に受理する形状
`parseContent()`（ノート読込 / `PUT /api/notes/:id` 経由の外部JSON）は、
`id` が文字列・`text` が文字列・`children` が配列、という**形（shape）だけ**を検査する。
**一意性・単一親・非循環・そもそも木かどうかは一切検査しない。** ノート内容はDBや APIから来る＝クライアントが形を作れるデータである。

### 反例（Alloyが見つけた「コードは受理するが木ではない」構造）
- `DuplicateIdHazard`: ルートと2つの子が**全て同じID**でも、形の検査は通る。
  この状態で「ID=dup のノードを削除」すると、`removeNode` は**最初の1つしか消さない**（2つ目が残る）。`findNode(dup)` は残った別ノードを返す。→ 削除したはずのものが残る／別物が編集される。
- `SharedChild`: 1つのノードが**2つの親**を持つ（＝木ではなくDAG）構造も受理される。編集すると `findParentAndIndex` が返す片方の親しか更新されず、もう片方の枝が静かに壊れる。
- `CodeAcceptsButNotTree`: 循環を含む構造すら形の検査は通る。

（`WellFormedTreeExists` で、正しい木も当然存在する＝宣言仕様自体は充足可能、を確認済み。）

### ドメイン的な影響
- 「各ノードをJSONとしてWeb公開」（TODO）や画像アップロード先の指定は**ノードIDでノードを一意に指す**ことに依存する。ID重複を許すと、公開・共有・削除の対象が定まらない。
- 壊れた（または悪意ある）ノートJSONを1つ保存されると、以後その文書のあらゆる編集操作が「最初の一致だけ」に効いて予測不能になる。

### 直し方の方向性
`parseContent()` に**木の健全性チェック**を入れる（読込時にIDの一意化＝`cloneWithNewIds`相当の正規化、単一親・非循環の検証、壊れていれば復旧 or 明示エラー）。

## 所見 3 — Undo/Redo後、選択が「存在しないノード」を指したままになりうる【重要度：中】

**モデル**: `formal/tla/MindMapUndo.tla`（TLA+ 状態機械）＋ `formal/z3/undo_redo.py`（Z3、実行可能なクロスチェック）
**再現**: FINDING 3（`reproduce.test.ts`）

### 何が宣言されているか
- `editorReducer.ts` 冒頭：「**常に1つのノードが選択されている（activeNodeId は null にならない）**」。
- `undoManager.ts`：Undoは「**ドキュメントだけを復元し、フォーカスは動かさない**」。
- `reconcileView()`：Undo/Redo後、選択ノードが消えていたら「ルートにフォールバックする」。

### 実際に保証されていること
- 型は `activeNodeId: string | null`（null を許す）。各操作は `if (!activeNodeId) return`（無効化）で自衛している。
- Undo/Redoは**ドキュメントだけ**を戻し、選択はそのまま。「選択が生きたノードを指す」不変条件は、**呼び出し側が別途 `reconcileView()` を呼ぶ**ことで初めて回復される（＝規約であって、型・状態として強制されていない）。

### 反例
- Z3（`undo_redo.py` パートB）：型・強制ルールを満たしたまま、`active` が**ドキュメントに存在しないノード**を指す状態を発見（例：ノードを1つ追加して選択→Undoで追加を取り消すと、選択だけが亡霊ノードを指す）。
- TLA+（`MindMapUndo.tla`、`ReconcileAuto=FALSE`）：`Init → AddNode → Undo` の短いトレースで不変条件 `ActiveAlwaysLive` が破れる。`ReconcileAuto=TRUE`（reconcileをUndo/Redoに畳み込む）にすると不変条件は保たれる＝これが修正案。
- Undo/Redoの**代数**（undo;redo=恒等、edit後はredo不可）自体は健全であることも Z3 で確認（パートA、深さ4まで反例なし）。

### ドメイン的な影響
`reconcileView()` を呼び忘れた／呼ぶ前のわずかな隙に、選択が亡霊ノードを指すと、`findNode` が null を返し、**キーボード操作がすべて静かに無反応**になる（同種の無反応バグは過去に修正実績あり：コミット `a6def5e`「Undoでactiveなノードが消えるとキー操作が無反応」）。

### 直し方の方向性
`reconcileView()` を「呼び出し側の別ステップ」ではなく、Undo/Redoの結果に**常に畳み込む**（TLA+ の `ReconcileAuto=TRUE` に相当）。合わせて、型を `string`（非null）に締めれば散在する `if (!activeNodeId)` 自衛コードも不要になる。

## 所見 4 — 行頭でEnterすると、ノードの**同一性（ID）が入れ替わる**【重要度：中／実行プローブ】

**再現**: FINDING 4（`reproduce.test.ts`。モデル検査ではなく実コード実行で確認）

子を持つノードの**先頭**でEnterを押すと：
```
P("Parent") ─ C("Child")     →     P("")           ← 元のノードは空になり子を失う
                                     新ID("Parent") ─ C("Child")   ← 本文と子は新IDへ
```
純粋なテキストアウトライナーとしては「上に空行が挿入された」ように見えて自然だが、edaneは**ノードIDを外部から参照する**設計（画像アップロード先、リンク、TODOの「ノード単位のWeb公開」）。「行を1つ足しただけ」の操作で本文の載るIDが変わるため、**そのノードを指す外部URL・参照が空ノードを指す**ようになりうる。

### 直し方の方向性
分割時に「本文＋子」を元ノード（元ID）側に残し、新規に作るのは空の兄弟にする（IDの意味＝コンテンツの同一性を保つ）。
