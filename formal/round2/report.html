所見 A — Undoが「見た目には無反応・次の1文字で帳消し」になる

A

Undoすると編集中ノードは取り消し前の本文を表示し続け、次の入力でUndoが取り消される

重要度: 高 · モデル: z3/view_faithfulness.py · tla/EditUndo.tla · 再現: round2.test.ts

宣言されている仕様

あらゆる通常操作は「編集バッファ editingText ＝ ノード本文」「カーソルは本文の範囲内」に そろえる（focusView）。画面もテキストエリアもこの editingText を描く—— 利用者が見る文字は、ドキュメントではなくビューのバッファそのもの。

コードが保証すること

reconcileView は「アクティブが存在すれば渡ってきたビューをそのまま返す」。 editingText・cursorPos はUndo前のドキュメントから作られた値のまま。 本文が縮んでもバッファもカーソルも追従しない。

反例 — Z3が発見し実コードで再現
ノードA = "hi there" (8文字) 編集中 · カーソル 8

① Aを "hi there" に編集中

→
ドキュメント: A = "hi" (2文字) 画面/入力欄: "hi there" · カーソル 8

② Cmd+Z。ドキュメントは戻るが表示は戻らない

何が起きるか

Undoが効いていないように見える。 編集中ノードの表示は editingText 駆動なので、 Cmd+Zを押しても画面は "hi there" のまま。実際にはドキュメントは "hi" に戻っている。

次の1文字で取り消しが帳消し。 入力欄には古い "hi there" が残っているので、"!" を足すと typeText が "hi there!" をモデルに再コミットする。戻したはずの本文が復活する。

カーソルが範囲外。 Z3の反例では バッファ2文字・本文1文字・カーソル2 —— カーソルが本文長を超える不整合な状態が、コードのルール上「合法」に到達可能だと示された。

ドメイン的な影響

Undoはアウトライン編集で最も信頼されるべき操作。それが無反応に見え、次の入力で失われると、 利用者は取り消しが消えたことに気づけない。過去にも近縁の無反応バグの修正実績がある （第1ラウンド所見3、コミット a6def5e）。

直し方の方向性

reconcileView を「存在確認」だけでなく、アクティブなノードの現在の本文から editingText を採り直し、カーソルを本文長にクランプするまで広げる。TLA+ 仕様では フラグ ReconcileRefreshesBuffer を FALSE→TRUE にすると不変条件 ViewFaithful が破れる状態から保たれる状態へ変わる（Z3の対照実験も同じくUNSAT＝健全になる）。

# python3 formal/round2/z3/view_faithfulness.py
SAT — counterexample view/document pair:
  active node          : 3  (present ✓ — reconcileView is satisfied)
  text(active) length  : 1   <- what the DOCUMENT says
  view.editingText len : 2   <- what the SCREEN + textarea show
  view.cursorPos       : 2
  (I1) violated: buffer shows 2 chars, node holds 1 — stale display.
  (I2) violated: caret at 2 > node length 1 — out-of-range caret.
  GAP CONFIRMED.
# contrast: reconcile re-derives buffer -> UNSAT (invariant holds) ✓

所見 B — 折りたたみの直後の編集で内容が消える

B

折りたたんだ節の直後で Backspace連結／Tabインデントすると、本文や選択中ノードが折りたたみの中へ消える

重要度: 中〜高 · モデル: z3/collapse_visibility.py · alloy/CollapseVisibility.als · 再現: round2.test.ts

宣言されている仕様

getFlatOrder は折りたたみの子孫を飛ばす（隠れたノードに着地しないため）。 コードは「触った内容を可視に保つ」を既に守っている——addChild・pasteBranch は挿入前に対象を開き、mergeSuccessorInto（行末Delete）は折りたたみに手を伸ばさない。

コードが実際にやること

mergeIntoPredecessor（Backspace連結）と indentNode（Tab）は 前の兄弟の折りたたみを一切見ず・開きもせずそのまま子を押し込む。5つの「下に入れる」経路で この2つだけが可視性を破る。

反例 B-1 — 行頭Backspaceの連結
ルート ├─ A ▸ (折りたたみ / A1が隠れる) └─ B ← 行頭でBackspace ├─ B1 └─ B2

意図: 「Bの行を前とつなげたい」

→
ルート └─ A ▸ ("AB" / 折りたたみのまま) ├─ B1 ✗ 不可視 └─ B2 ✗ 不可視

flatOrder = [root, A] · B1・B2が画面から消失

反例 B-2 — Tabインデント（選択中ノードが消える）
ルート ├─ A ▸ (折りたたみ) └─ B ← 選択中。Tab

意図: 「Bを一段下げたい」

→
ルート └─ A ▸ (折りたたみ) └─ B ✗ 不可視（だが選択中）

flatOrder = [root, A] · 選択中のノード自身が消える

ドメイン的な影響

折りたたみは長い文書を畳んで扱う中心機能で、「折りたたんだ節の直後の行」は日常的に生じる。 そこで多用される Backspace連結／Tabインデントが、内容や選択を静かに折りたたみの中へ隠す。 利用者は削除した覚えがないのに小項目が消えたように見え、しかも addChild／paste では 起きないため挙動が一貫しない。所見B-2は「選択が見えないノードを指す」状況（第1ラウンド所見3）の別経路での再発。

直し方の方向性

backspaceAtStart／tab(indent) の連結・移動先が折りたたまれているとき、 addChild／pasteBranch と同じく対象を開いてから入れる （toggleCollapse(target, false)）。行末Delete mergeSuccessorInto が既に持つ 「折りたたみに手を伸ばさない」判断と対称にそろえるのが素直。
