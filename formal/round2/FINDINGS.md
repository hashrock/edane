## 所見 A — Undoすると、編集中のノードは「取り消し前の本文」を表示し続け、次の1文字でUndoが帳消しになる【重要度：高】

**モデル**: [`z3/view_faithfulness.py`](./z3/view_faithfulness.py)（実行可、反例をZ3が探索）＋
[`tla/EditUndo.tla`](./tla/EditUndo.tla)（状態機械・不変条件）
**再現**: FINDING A（`round2.test.ts`）

### 何が宣言されているか
- `editorReducer.ts` の `focusView()` をはじめ、**あらゆる通常操作**は、フォーカスを移すと
  「編集バッファ（`editingText`）＝そのノードの本文」「カーソルは本文の範囲内」に**必ず**そろえる。
  これがreducer自身の約束事：**画面に出る文字とカーソルは、常にドキュメントのそのノードと一致する。**
- 画面描画もこれに乗っている：アクティブなノードのキャンバス表示は `editingText` から描き
  （`MindmapEditor.tsx` 1155・1204行）、テキストエリアも `el.value = editingText; setSelectionRange(cursorPos,…)`
  で駆動する（427-428行）。**つまり利用者が見る文字・カーソルは、ドキュメントではなくビューのバッファそのもの。**

### コードが実際に保証していること
第1ラウンドの所見3の修正で、Undo/Redoが通る `replace` 経路に `reconcileView()` を畳み込み、
**「アクティブなノードは必ず存在する」**は強制されるようになった。しかし `reconcileView()` は：

```ts
if (view.activeNodeId && findNode(model, view.activeNodeId)) return view; // ← 生きていれば「そのまま」返す
```

アクティブなノードが**生きてさえいれば、渡ってきた（取り消し前の）ビューをそのまま返す**。
`editingText` と `cursorPos` は、**別のドキュメント（Undo前）から作られた値のまま**コピーされる。
`restoreDocument()` は現在のビュー＋復元後ドキュメントを `replace` に渡すので、
**バッファとカーソルはUndoに追従しない。**

### 反例（Z3が発見・実コードで再現）
Z3が「アクティブは存在する（コードの保証を満たす）が、バッファ長≠本文長・カーソルが範囲外」
の状態を発見。実コードでの具体例：

```
ノードA の本文 = "hi there"（8文字）を編集中（カーソル8）
        │  Cmd+Z（Undo）→ ドキュメントは A="hi"（2文字）に復元される
        ▼
画面／テキストエリアは "hi there" のまま・カーソルは8（本文長2を超過）
```

- **利用者から見るとUndoが効いていない**：編集中ノードの表示は `editingText`（＝"hi there"）駆動なので、
  Cmd+Zを押しても画面は "hi there" のまま変わらない。実際にはドキュメントは "hi" に戻っている。
- **次に1文字打つと取り消しが帳消し**：テキストエリアには古い "hi there" が残っているので、
  そこに "!" を足すと `typeText` が **"hi there!" をモデルに再コミット**する。Undoで戻したはずの本文が
  そっくり復活する（`round2.test.ts` の FINDING A 2本目で実証）。
- **カーソルが範囲外**：`cursorPos=8` は本文長2を超える（Z3の反例：バッファ2文字・本文1文字・カーソル2）。

### ドメイン的な影響
Undoはアウトライン編集で最も信頼されるべき操作。それが「見た目には無反応・次の入力で無かったことになる」
と、利用者は**自分の取り消しが失われたことに気づけない**。過去にも近縁の無反応バグの修正実績がある
（第1ラウンド所見3、コミット `a6def5e`）。第1ラウンドの修正は「亡霊ノードを指す」問題は封じたが、
**「生きたノードだが本文・カーソルが古い」問題は残った**＝不変条件を “存在” までしか締めていなかった。

### 直し方の方向性
`reconcileView()` を「アクティブの存在確認」だけでなく、**アクティブなノードの現在の本文から
`editingText` を採り直し、`cursorPos`/`selectionEnd` を本文長にクランプする**まで広げる
（TLA+ の `ReconcileRefreshesBuffer=TRUE` に相当。同フラグ `FALSE` で不変条件 `ViewFaithful` は破れ、
`TRUE` で保たれることを仕様で示した。Z3の対照実験でも同じくUNSAT＝健全になる）。
これで「ビューは常にアクティブなノードを忠実に映す」不変条件が、通常操作だけでなくUndo/Redoでも
reducer自身によって強制される。

## 所見 B — 折りたたんだ節の直後で「行の連結（Backspace）／インデント（Tab）」すると、本文や選択中ノードが折りたたみの中へ消える【重要度：中〜高】

**モデル**: [`z3/collapse_visibility.py`](./z3/collapse_visibility.py)（実行可）＋
[`alloy/CollapseVisibility.als`](./alloy/CollapseVisibility.als)（Alloyが構造反例を発見・実行済み）
**再現**: FINDING B（`round2.test.ts`）

### 何が宣言されているか
- `getFlatOrder()`：折りたたまれた（collapsed）ノードの子孫は飛ばす。「**キーボード操作が隠れた
  ノードに着地しない**ように」。ノードの可視性＝「折りたたまれた祖先を持たない」こと。この一覧が、
  全ナビゲーションとカーソル移動が歩く順序。
- コードは**「触った直後の内容を可視に保つ」ことを既に不変条件として扱っている**：
  - `addChild` … 子を足す前に `toggleCollapse(model, id, false)`（対象を**開いてから**追加）
  - `pasteBranch` … 貼り付け前に同じく対象を開く
  - `mergeSuccessorInto`（行末Delete）… `!node.collapsed` で**折りたたみの中には手を伸ばさない**
- ⇒ 宣言／観測される不変条件 **(V): 構造編集は、連結・インデントした内容を折りたたみの中に隠さない。**

### コードが実際にやること
一方、前の兄弟の下へノードを移す2つの操作は、**対象の折りたたみを一切見ない・開きもしない**：
- `mergeIntoPredecessor()`（行頭Backspaceの連結先）：前の兄弟Pがあると `P.children.push(...node.children)`。
  **Pが折りたたまれていても**そのまま子を突っ込む。
- `indentNode()`（Tab）：`prevSibling.children.push(node)`。**前の兄弟が折りたたまれていても**そのまま。

`addChild`/`pasteBranch` の「開いてから入れる」も、`mergeSuccessorInto` の `!collapsed` ガードも**通らない**。
5つある「下に内容を入れる」経路のうち、**この2つだけが可視性の不変条件を破る**。

### 反例（ドメインの言葉で）
**B-1: 行頭Backspaceの連結**
```
ルート
├─ A ▸（折りたたみ済み。中にA1が隠れている）
└─ B          ← ここ（"B"の行頭）でBackspace
    ├─ B1
    └─ B2
```
利用者の意図：「Bの行を前とつなげたい」。実際の結果（実コードで再現）：
```
ルート
└─ A ▸（"AB" に連結・折りたたみのまま）   ← B1・B2 はAの子になったが折りたたみの中で不可視
```
`getFlatOrder` は `["root","A"]` になり、**B1・B2 は画面からもキーボード操作からも消える**。
カーソルは折りたたまれたAに着地する。**1回のBackspaceで、見えていた小項目2つが行方不明になる。**

**B-2: Tabインデント**
```
ルート
├─ A ▸（折りたたみ済み）
└─ B          ← 選択中。ここでTab
```
結果：Bは折りたたまれたAの子になり不可視化されるが、**Bは依然として「選択中（activeNodeId）」のまま**。
`getFlatOrder` は `["root","A"]`。**選択中のノードそのものが画面から消える**（第1ラウンド所見3で修正した
「選択が見えないノードを指す」状況が、別経路で再発する）。

Alloyは5ノード探索で「移された内容が折りたたまれた対象の直下にあり不可視」の構造を自動発見
（`JoinHidesContent` → SATISFIABLE）。対照の `ExpandFirstNeverHides`（対象を開いてから入れる＝
addChild/paste）は **UNSAT**（隠せない）で、この2操作だけが例外であることを裏づけた。

### ドメイン的な影響
折りたたみは長い文書を畳んで扱うための中心機能で、「折りたたんだ節の直後の行」は日常的に生じる。
そこで最も多用されるBackspace連結／Tabインデントが、**内容や選択を静かに折りたたみの中へ隠す**。
利用者は削除した覚えがないのに小項目が消えたように見え、`addChild`/`paste` では起きないため
挙動が一貫しない。

### 直し方の方向性
`backspaceAtStart`／`tab(indent)` の連結・移動先が折りたたまれている場合に、
`addChild`/`pasteBranch` と同じく**対象を開いてから**入れる（`toggleCollapse(target, false)`）。
あるいは連結・インデント後に、移した内容（Bの子／B自身）が `getFlatOrder` に載ることを
不変条件として reducer 側で保証する。行末Deleteの `mergeSuccessorInto` が既に持つ
「折りたたみに手を伸ばさない」判断と対称にそろえるのが素直。
